# นโยบายเก็บ/ลบข้อมูล (Data Retention & Soft-delete)

หลักการ: **ข้อมูลการเงินและยอดขายห้ามลบถาวร** เพื่อให้ตรวจสอบย้อนหลัง/ปิดบัญชี/ภาษีได้

## สถานะปัจจุบันในระบบ

| ข้อมูล | วิธีจัดการ | หมายเหตุ |
|--------|-----------|----------|
| เมนู (`menus`) | soft-delete ด้วย `is_archived` | ไม่ลบจริง — บิลเก่ายัง snapshot ชื่อ/ราคา/รูปไว้แล้ว |
| บิล (`bills`) | ไม่ลบ; คืนเงินใช้สถานะ `refunded` | ยอดขาย/ใบกำกับภาษีคงอยู่เสมอ |
| รายการอาหาร (`order_items`) | ไม่ลบ; ยกเลิกใช้สถานะ `voided` + เหตุผล | คงประวัติการยกเลิก |
| การกระทำสำคัญ (`audit_logs`) | append-only | snapshot ชื่อผู้ทำ คงอยู่แม้ลบพนักงาน |
| พนักงาน (`staff`) | ลบได้จริง | audit/บิล เก็บ snapshot ชื่อไว้แล้ว (ประวัติไม่หาย) |
| สมาชิก (`members`) | ลบได้ (FK `SET NULL` บนบิล) | บิลเก่าไม่พังเมื่อสมาชิกถูกลบ |

## แนวทางที่แนะนำ (ตั้งเป็นนโยบาย)

1. **ห้ามลบ** `bills`, `order_items`, `shifts`, `audit_logs`, `reservations` — เก็บถาวร (อย่างน้อย 5 ปีตามอายุความภาษีไทย)
2. **Soft-delete** สำหรับ entity ที่ "เลิกใช้" แต่ถูกอ้างอิงในประวัติ (เช่นเมนู) — ใช้ flag ไม่ลบแถว
3. **สำรองข้อมูล** เป็น safety net จริง — ดู [`backup-and-restore.md`](./backup-and-restore.md) (รายวัน + เก็บ 30 วัน)
4. ปิดร้าน (offboard) → ตั้ง `shops.status` เป็นสถานะปิด/เก็บถาวร **ไม่ลบ record** เพื่อคงยอดย้อนหลัง
   - หมายเหตุ: ปัจจุบัน `ShopStatus` มีแค่ `pending`/`active` — ถ้าต้องการสถานะ `archived` ค่อยเพิ่ม enum + migration ภายหลัง

## PDPA (ข้อมูลส่วนบุคคล)
- ข้อมูลส่วนบุคคลที่เก็บ: ชื่อ/เบอร์โทรของ **สมาชิก** และ **การจอง**
- ถ้าลูกค้าขอให้ลบ: ลบ/ปกปิด field ชื่อ-เบอร์ ของ `members`/`reservations` ได้ โดย **คงยอดบิล** (บิลผูกผ่าน id ไม่ต้องเก็บ PII)
